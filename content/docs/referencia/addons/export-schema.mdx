# exportSchema

A camada de **exportação** do app. Você desenha uma árvore de propriedades (objeto/array/valor), bind cada folha a um campo real do projeto, e o app gera o JSON que o motor consome — com refs cross-section resolvidas e formatos de array configuráveis.

## Por que importa

Sem exportSchema, você precisaria escrever um exporter custom: ler addons, seguir refs, formatar arrays, montar JSON. Cada projeto reinventaria isso. Aqui você desenha o JSON visualmente — e a saída é deterministicamente reproducível.

E a importação **round-trip** (re-aplicar JSON exportado de volta no projeto) torna possível workflows tipo "designer ajusta valores no JSON exportado pro Unity, depois reimporta no GDD pra manter doc atualizado".

## Onde costuma viver

- Página dedicada de tipo `📤 Remote Config` ou em branco.
- Próxima de páginas que vão ser exportadas (ex.: junto com Skills).

**Não é singleton** — uma página pode ter múltiplos exportSchemas se quiser produzir múltiplos arquivos JSON.

## `ExportSchemaNode`

Um nó é um dos três tipos:

### Objeto

```ts
{
  nodeType: "object",
  key: "minha_chave",
  children: ExportSchemaNode[]
}
```

Renderiza como `{ "minha_chave": { ...children } }`.

### Array

```ts
{
  nodeType: "array",
  key: "lista",
  arraySource: ArraySource,
  itemTemplate: ExportSchemaNode[]
}
```

`arraySource` decide o que iterar (uma tabela, um craftTable, um skills addon, etc.). `itemTemplate` é o esqueleto aplicado a cada item.

### Valor

```ts
{
  nodeType: "value",
  key: "campo",
  binding: ExportSchemaBinding,
  abs?: boolean,         // valor absoluto ao final (number)
  multiplier?: number    // multiplica o valor final (number)
}
```

`binding` decide de onde vem o valor (manual, dataSchema, rowColumn, skillField, etc.).

## Sources de array

Cada source diz **o que iterar**:

| Source | O que itera |
|---|---|
| `progressionTable` | Linhas de uma tabela de progressão |
| `craftTable` | Entries (receitas) de uma mesa |
| `productionIngredients` | Ingredientes do production atual (dentro de craftTable) |
| `productionOutputs` | Outputs do production atual |
| `skills` | Entries de um skills addon |
| `skillCosts` | Custos da skill atual (dentro de skills) |
| `skillEffects` | Efeitos da skill atual (dentro de skills) |

Sources contextuais (`skillCosts`, `skillEffects`, `productionIngredients/Outputs`) **só funcionam dentro de um array pai compatível**. O picker da UI esconde opções inválidas.

## Bindings de valor

Cada source tem **bindings** correspondentes pra acessar campos:

| Binding | De onde lê | Exemplos de campo |
|---|---|---|
| `manual` | Constante hardcoded | qualquer string/number/boolean |
| `dataSchema` | Entry específica de um dataSchema | qualquer entry.value |
| `rowLevel` | Level da linha atual da progressionTable | `1`, `2`, ... |
| `rowColumn` | Coluna específica da linha | qualquer coluna |
| `entryField` | Campo escalar do craftTable entry atual | `productionRef`, `category`, `unlock.*` |
| `productionField` | Segue productionRef → lê campo | `name`, `mode`, `craftTimeSeconds` |
| `itemField` | Item da iteração ingredients/outputs | `itemRef`, `quantity` |
| `skillField` | Campo da skill atual | `id`, `name`, `kind`, `cooldownSeconds`, `tagsCsv`, `unlock.*` |
| `skillCostField` | Campo do cost atual | `id`, `type`, `amount`, `currencyRef`, `attributeKey` |
| `skillEffectField` | Campo do effect atual + resolved fields | `resolvedName`, `resolvedAttributeKey`, `resolvedDefinitionsRef`, `resolvedValue`, `resolvedDurationSeconds`, `resolvedTickIntervalSeconds`, `resolvedStacking`, etc. |

**Refs viram dataIds**: campos como `productionRef`, `currencyRef`, `unlockItemRef`, `attributeModifiersSectionId` resolvem pro `dataId` da seção alvo (ex.: `DATA_FIRE_BALL`) — não pro UUID interno. JSON exportado fica humano e estável.

## Formatos de array

Quando o nó é array de objetos com `level`, você escolhe como serializar:

### `rowMajor` (padrão)

```json
[
  {"level": 1, "price": 100, "cap": 10},
  {"level": 2, "price": 200, "cap": 20}
]
```

### `columnMajor`

```json
{
  "level": [1, 2, 3],
  "price": [100, 200, 300],
  "cap": [10, 20, 30]
}
```

Útil quando o motor lê linha por coluna (ex.: gráficos).

### `keyedByLevel`

```json
{
  "1": {"price": 100, "cap": 10},
  "2": {"price": 200, "cap": 20}
}
```

Útil quando o motor usa `data[level]` direto.

### `matrix`

```json
{
  "headers": ["level", "price", "cap"],
  "rows": [[1, 100, 10], [2, 200, 20]]
}
```

Compacto. Útil pra CSVs e tabelas binárias.

## Exemplo: skills export

```json
{
  "id": "rc-skills",
  "name": "Skills Export",
  "arrayFormat": "rowMajor",
  "nodes": [
    {
      "nodeType": "array",
      "key": "skills",
      "arraySource": { "type": "skills", "addonId": "section-fire-ball" },
      "itemTemplate": [
        { "nodeType": "value", "key": "id",   "binding": { "source": "skillField", "field": "id" } },
        { "nodeType": "value", "key": "name", "binding": { "source": "skillField", "field": "name" } },
        {
          "nodeType": "array",
          "key": "effect",
          "arraySource": { "type": "skillEffects" },
          "itemTemplate": [
            { "nodeType": "value", "key": "attr_id",    "binding": { "source": "skillEffectField", "field": "resolvedDefinitionsRef" } },
            { "nodeType": "value", "key": "attr_key",   "binding": { "source": "skillEffectField", "field": "resolvedAttributeKey" } },
            { "nodeType": "value", "key": "attr_value", "binding": { "source": "skillEffectField", "field": "resolvedValue" } }
          ]
        }
      ]
    }
  ]
}
```

Resultado:

```json
{
  "skills": [
    {
      "id": "DATA_FIRE_BALL",
      "name": "Fireball",
      "effect": [
        { "attr_id": "DATA_BASIC_ATTR", "attr_key": "hp", "attr_value": -10 }
      ]
    }
  ]
}
```

## Importação round-trip

O app aceita **importar** um JSON exportado e re-aplicar os valores no projeto. Isso permite:

- Designer edita JSON exportado pra Unity, reimporta no GDD pra sincronizar
- Backup/restore parcial
- Versioning de balanceamento via git no JSON

Limitação: estruturas complexas (refs cross-section) podem não importar 100% — mas valores simples (numbers, strings, booleans) sim.

## Wizard

Não tem wizard próprio — você adiciona o addon e desenha a árvore manualmente. O app guia via UI (sources contextuais, bindings filtrados, preview ao vivo).
